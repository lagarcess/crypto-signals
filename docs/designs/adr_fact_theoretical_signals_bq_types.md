# ADR: BigQuery Column Types for fact_theoretical_signals Nested Fields

## Status
Accepted

## Context
The `fact_theoretical_signals` table replaces fragmented tables (like `fact_rejected`, `fact_expired`) as the singular, unified quantitative backtesting sandbox. It must capture the complete `Signal` Pydantic structure generated by `signal_generator.py`, including nested fields related to the lifecycle of the signal.

When migrating from Firestore (NoSQL) to BigQuery (SQL), we face impedance mismatches for deeply nested, variable-length, or dynamic structures. The three main nested components of `FactTheoreticalSignal` that need explicit type mapping to BigQuery are:
1. `confluence_factors`: A map of technical indicators evaluated at signal generation.
2. `exit_reasons`: A list of factors or triggers leading to the simulated or factual exit.
3. `drawdown_metrics`: Structured metrics reflecting the maximum adverse excursion of the trade.

## Decision

We will implement the following structured mappings in BigQuery:

### 1. `confluence_factors` -> `JSON`
- **Pydantic Model**: `Dict[str, float]` (or `Dict[str, Any]`)
- **Reasoning**: The keys inside `confluence_factors` vary significantly between strategies (e.g., `RSI_14` for Mean Reversion vs. `MACD_hist` for Trend Following). A rigid BigQuery `STRUCT` (RECORD) cannot gracefully handle unknown/dynamic keys without schema breaking. The `JSON` datatype allows dynamic keys to be queried directly using `JSON_EXTRACT_SCALAR`.

### 2. `exit_reasons` -> `REPEATED STRING` (Array)
- **Pydantic Model**: `List[str]`
- **Reasoning**: A signal might have multiple compounding factors leading to its exit (e.g., `["TrailingStopHit", "SessionClose"]`). BigQuery handles string arrays intuitively using the `REPEATED` mode.

### 3. `drawdown_metrics` -> `RECORD` (Struct)
- **Pydantic Model**: A dedicated `DrawdownMetrics` Pydantic BaseModel (containing `max_dd_pct: float` and `duration_hours: int`).
- **Reasoning**: Unlike confluence factors, drawdown metrics represent a strictly defined, predictable set of keys. Mapping this as a BigQuery `RECORD` provides the strongest type-safety without resorting to weakly-typed JSON parsing downstream.

## Consequences
- BigQuery table creation scripts must utilize `JSON` fields, `ARRAY<STRING>`, and custom `STRUCT`s.
- The Python Google Cloud BigQuery client's `insert_rows_json` handles JSON serialization automatically if the Pydantic dict dumps correctly; however, specific recursive flattening might be needed if Pydantic passes complex types.
- Future analytics on `confluence_factors` will require `JSON_EXTRACT` syntax in SQL.
